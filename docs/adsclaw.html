AdsClaw — visão simples

Um “piloto automático” de performance com humano no controle.

O AdsClaw é um agente que monitora contas de anúncios, detecta problemas (ex.: fadiga criativa), gera ações (copy/creatives) e pede aprovação quando necessário — tudo registrado no Supabase e visível no Cockpit.

Entradas: Telegram · HTTP · Cron Cérebro: AgentLoop (ReAct) Ações: Tools (Meta/Google/Creative) Memória: Supabase chat_history UI: Cockpit (Dashboard/Alertas/Conversas)
Como pensar no sistema
O AdsClaw faz bem
monitorar → decidir → agir
Tudo via tools: auditável, repetível.
Onde ele “não inventa”
métricas reais (Meta/Google)
Integrações via MCP servers.
Onde o humano entra
aprovações e decisões
Human-in-the-loop (Telegram/Cockpit).
Onde tudo fica salvo
Supabase (DB + Storage)
Histórico, alertas, assets e regras.

Como funciona (bem simples)

O fluxo abaixo resume o “motor” do AdsClaw. A parte importante: o modelo decide quando chamar uma tool, o sistema executa, guarda a observação e continua até terminar. 

Entradas
  ├─ Telegram (mensagens / botões)
  ├─ HTTP (/api/*)
  └─ Orchestrator (cron: auditorias)
         │
         ▼
SkillRouter (escolhe “persona/skill” do agente)
         │
         ▼
AgentLoop (ReAct)
  ├─ pensa (Thought)
  ├─ escolhe uma Tool (Action)
  ├─ executa a Tool (Observation)
  └─ repete até concluir (ou limite)
         │
         ├─ Supabase: salva memória em chat_history
         ├─ Supabase: cria/atualiza alerts e pending_approvals
         └─ Storage: publica assets (imagens/vídeos)
         ▼
Saídas
  ├─ Resposta no Telegram/HTTP
  └─ Cockpit (Dashboard, Alertas, Aprovações, Conversas)

Pastas típicas no repo: agent/ (backend do agente), cockpit/ (frontend), mcp-servers/ (integrações), supabase/ (schema/migrations).

Cases de uso (na prática)

Monitorar performance Meta/Google

Consulta métricas reais (CPA/ROAS/CTR/etc.), compara com as regras do cliente e gera insights acionáveis.

Detectar fadiga criativa alerta

Se o refresh de criativos passou do prazo, cria um alerta e inicia o fluxo de decisão (“o que fazer agora?”).

Laboratório Criativo gerar

Gera copies, imagens e vídeos (baseado em benchmark e contexto do cliente) e salva no Storage.

Aprovação do gestor HITL

Antes de ações sensíveis, cria uma pending approval e notifica o gestor para aprovar/rejeitar.

Notificações no Telegram notify

Envia alertas e “resumos executivos” com botões inline e histórico rastreável por cliente.

Cockpit para operação UI

Um painel único para ver KPIs, alertas, aprovações pendentes e as conversas anteriores do agente.

Tools e Skills (explicação simples)

No AdsClaw existem dois conceitos que parecem iguais, mas não são:

Tools (runtime do agente) executam ações

São funções registradas no ToolRegistry para o AgentLoop chamar. Elas fazem trabalho real (Supabase, MCP, geração, notificações).

  • Dados e regras: list_clients, get_client_rules
  • Meta Ads (MCP): meta_list_campaigns, meta_get_account_insights, meta_get_campaign_insights
  • Google Ads (MCP): google_list_campaigns, google_get_performance
  • Benchmarks: fetch_ad_benchmarks
  • Copy e mídia: generate_ad_copy, generate_image, generate_video
  • Assets: upload_asset
  • Human-in-the-loop: ask_approval, notify_manager

Skills (dev/Cursor) ajudam a construir

São “habilidades” do assistente durante o desenvolvimento (documentação, UI, testes, etc.). Elas não são tools do runtime.

  • Planejar e esclarecer: brainstorming / SDD (Spec-Driven Development)
  • UI: frontend-design (componentes e páginas)
  • Testar: webapp-testing / Playwright (fluxos no cockpit)
  • Integrações: mcp-builder (criar MCP servers)
  • Conteúdo: doc-coauthoring (docs, guias e specs)

Dica: se você está operando o AdsClaw em produção, pense em tools. Se você está evoluindo o produto, pense em skills.
